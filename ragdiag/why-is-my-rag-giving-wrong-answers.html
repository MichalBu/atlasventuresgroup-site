Why is my RAG giving wrong answers?

"The RAG is hallucinating" is a symptom, not a diagnosis. A retrieval-augmented pipeline has at least six places a wrong answer can originate, and the fix for each is different - re-chunking your corpus won't help if the reranker is burying the right passage. Here's the symptom-to-component map, with a quick manual test for each.

Symptom: the answer mixes up facts from different documents

Likely component: chunking. Chunks that cross document or section boundaries let the model blend unrelated contexts (the classic cross-document confusion failure). Quick test: print the exact retrieved chunks for one bad query - if a single chunk contains text from two sources or two topics, it's chunking.

Symptom: retrieved documents are on-topic but never quite the right one

Likely component: embedding model. The embedding space groups by broad topic but can't separate near-neighbors (e.g. v2 vs v3 of the same doc, similar products). Quick test: embed the query and the known-correct passage; if their similarity is lower than the query's similarity to the wrong-but-related passage, the embedding can't make this distinction.

Symptom: the right passage exists but isn't in the retrieved set

Likely component: retrieval ranking / top-k. Quick test: query the vector store directly with k=50 and search for the correct passage's rank. Rank 12 with k=5 is a ranking/k problem; absent at k=50 points back to chunking or embedding.

Symptom: raising k made answers worse, not better

Likely component: reranker (or its absence). More candidates only help if something orders them well; without a reranker, extra passages are extra noise for the generator. Quick test: compare answer quality at k=5 vs k=20 with the reranker toggled on and off on the same query set.

Symptom: rephrasing the question changes the answer completely

Likely component: query preprocessing. Abbreviations, typos, or multi-part questions that a query-rewrite step should be normalizing. Quick test: log the final string actually sent to retrieval; if it's the user's raw text, this stage doesn't exist yet.

Symptom: the correct passage was retrieved and the answer is still wrong

Likely component: generation. The one case that isn't retrieval's fault: context window overload, a prompt that doesn't force grounding, or the model overriding context with its own priors. Quick test: hand the model only the correct passage and the question; if it still answers wrong, it's prompt/model, not pipeline.

The manual tests above, automated per failure

Each test above is a counterfactual: change one component, see if the answer flips.
